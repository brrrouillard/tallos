# Tallos

Tallos is a **B2B platform where teams spin up AI agents to help them get work done**. Agents are powered by **Hermes** behind the scenes.

- **MVP (current):** a single built-in agent — a productivity agent.
- **v2 (planned):** users dynamically create and configure their own agents.

Design implication: keep the agent layer **data-driven** from the start. The MVP ships one fixed agent, but model agent definitions/config as data so the move to many user-defined agents in v2 doesn't require a rewrite. Avoid hardcoding "the productivity agent" into shared infrastructure.

## Stack

- **Bun** — package manager, script runner, and test runner. Also the preferred runtime for any server-side code.
- **TanStack Start** — full-stack React framework, SSR served by **Nitro 3**.
- **React 19** + **Tailwind CSS v4** + **shadcn/ui** (radix-luma preset: Radix primitives, lucide icons).
- **Vite 8** — bundler/dev server (`@tanstack/react-start/plugin/vite`, `@vitejs/plugin-react`, `nitro/vite`, `@tailwindcss/vite`).
- **TypeScript 7** — the native compiler (currently `7.0.1-rc`).
- **oxlint + oxfmt**, configured via **ultracite** (strict ruleset).

> The frontend intentionally uses **Vite + TanStack Start**. Do **not** convert it to `Bun.serve()` + HTML imports — that earlier guidance no longer applies. Bun stays as the package manager, script runner, and server-side runtime.

## Commands

```bash
bun install         # install dependencies
bun run dev         # dev server (vite dev) on http://localhost:3000
bun run build       # production build (vite build → Nitro output in .output/)
bun run start       # run the built server (bun .output/server/index.mjs)
bun run check       # lint + format check (ultracite → oxlint + oxfmt), no writes
bun run fix         # auto-fix lint + formatting
bunx tsc --noEmit   # typecheck with TS 7
bun test            # run tests
```

Scope `check`/`fix` to changed paths to avoid reformatting unrelated files, e.g. `bunx ultracite fix src vite.config.ts`.

## Project structure

```
src/
  router.tsx          # exports getRouter() — TanStack Start auto-discovers it
  routes/             # file-based routes; adding a file regenerates routeTree.gen.ts
    __root.tsx        # document shell (html/head/body); <head> meta + app.css link
    index.tsx         # "/" route
  routeTree.gen.ts    # GENERATED by tanstackStart() — do not edit (committed to repo)
  styles/app.css      # Tailwind entry + shadcn theme tokens (light/dark, @theme inline)
  lib/utils.ts        # cn() helper (clsx + tailwind-merge)
  components/ui/      # shadcn components land here after `bunx shadcn@latest add ...`
vite.config.ts        # plugin order matters: tanstackStart() before viteReact()
components.json       # shadcn config (aliases on ~, css → src/styles/app.css)
oxlint.config.ts      # extends ultracite/oxlint/core
oxfmt.config.ts       # extends ultracite/oxfmt
```

- Import alias: `~/*` → `src/*` (resolved by Vite's native `resolve.tsconfigPaths`, no `vite-tsconfig-paths` plugin).
- Routing is file-based — `routeTree.gen.ts` is regenerated on `dev`/`build`; never hand-edit it.

## Code style (oxlint via ultracite is strict)

New code must pass `bun run check`. The rules that bite most often:

- Write functions/components as **arrow expressions** (`const X = () => …`), not `function` declarations.
- **Sort object keys alphabetically** (router options, config objects, JSX prop objects). Array element order is unaffected.
- Define a component/function **before** it is referenced (no-use-before-define).
- Use a separate `import type { … }` line, not inline `import { type … }`.
- shadcn/Tailwind: semantic tokens (`bg-background`, `text-muted-foreground`) over raw colors; `cn()` for conditional classes; `gap-*` not `space-*`; `size-*` when width == height.

`*.gen.*` (e.g. `routeTree.gen.ts`) is already auto-ignored by oxlint/oxfmt. shadcn files in `src/components/ui/**` use shadcn's own style and will trip the strict rules — when you add components, either conform them or add `"src/components/ui/**"` to `ignorePatterns` in both `oxlint.config.ts` and `oxfmt.config.ts`.

## Server-side code (Bun)

Prefer Bun built-ins over third-party packages on the server:

- `bun:sqlite` (not better-sqlite3) · `Bun.sql` for Postgres (not pg / postgres.js) · `Bun.redis` (not ioredis).
- `Bun.file` over `node:fs` read/write · `Bun.$` (shell) over execa · built-in `WebSocket` / `fetch`.
- Bun auto-loads `.env` — no dotenv.

## Agents (Hermes)

Agent capabilities are backed by **Hermes**. There is no agent code in the repo yet — when adding it, keep agent definitions configurable/data-driven (see the design implication above) so the MVP's single productivity agent generalizes cleanly to user-created agents in v2.
